What Is an Exception?

~10 min

When Python encounters something it can't do — dividing by zero, opening a missing file, converting a word to a number — it raises an exception and stops executing. Without handling, your program crashes.

# Run this — what happens?
int("dragon")      # ValueError: invalid literal for int()
10 / 0              # ZeroDivisionError: division by zero
my_list[99]        # IndexError: list index out of range

Exceptions aren't bugs — they're Python's way of communicating what went wrong. Our job is to catch them so the game (or program) keeps running.

ExceptionTriggered When…Example
ValueErrorRight type, wrong valueint("hello")
TypeErrorWrong type entirely"hp" + 100
ZeroDivisionErrorDividing by zeroscore / 0
IndexErrorList index doesn't existparty[10]
KeyErrorDict key not foundstats["mana"]
FileNotFoundErrorFile path doesn't existopen("save.txt")
NameErrorVariable not definedprint(dragon)

try / except — The Basics

~10 min

Wrap risky code in a try block. If an exception fires, Python jumps to the matching except block instead of crashing.

try: (risky code runs here)
↓
exception raised?
↓ yes
except ExceptionType: (handle it)
↓
program continues normally
# Basic structure
try:
    raw = input("Enter your hero's level: ")
    level = int(raw)          # could raise ValueError
    print(f"Level {level} hero ready!")
except ValueError:
    print("Invalid! Please enter a number.")

# Catching ANY exception (use sparingly — too broad)
try:
    risky_operation()
except Exception as e:
    print(f"Something went wrong: {e}")
⚠️ Avoid bare except: — It catches everything, including keyboard interrupts and system exits. Always name at least Exception.

Multiple except Clauses

~10 min

You can handle different exceptions differently by stacking except blocks. Python tries them top to bottom — first match wins.

def get_party_member(party, index):
    try:
        index = int(index)        # ValueError if not a number
        return party[index]        # IndexError if out of range
    except ValueError:
        print("Index must be a number.")
    except IndexError:
        print("No party member at that slot.")

party = ["Aria", "Brom", "Caela"]
get_party_member(party, "two")   # ValueError
get_party_member(party, 99)     # IndexError

# Catching multiple in one line:
except (ValueError, TypeError):
    print("Bad input type or value.")

else and finally

~10 min

The full try structure has four optional clauses. else runs only when no exception occurred. finally runs no matter what.

try: risky code
↓ exception?
except:
yes → handle
else:
no → success code
↓ always
finally: cleanup (always runs)
def load_save_file(filename):
    try:
        f = open(filename)
        data = f.read()
    except FileNotFoundError:
        print("Save file not found — starting fresh!")
    else:
        print(f"Loaded {len(data)} bytes.")  # only if open() succeeded
    finally:
        print("Load attempt complete.")  # always
💡 Rule of thumb: Put success logic in else (not in try) to keep the try block as small as possible — only the line that might fail goes in there.

Raising Exceptions with raise

~10 min

You can raise your own exceptions to enforce rules — for example, rejecting impossible HP values before they cause weird bugs later.

# Raise a built-in exception with a custom message
def set_hp(value):
    if value < 0:
        raise ValueError(f"HP cannot be negative: {value}")
    if type(value) is not int:
        raise TypeError("HP must be an integer.")
    return value

# The caller handles it:
try:
    hero_hp = set_hp(-10)
except ValueError as e:
    print(f"Error: {e}")
    hero_hp = 0  # safe default
Pattern: Functions raise exceptions to signal invalid input. The calling code catches them and decides what to do. This keeps validation logic in one place.

From Loops to Comprehensions

~12 min

A list comprehension builds a new list in a single, readable expression. Any for loop that appends to an empty list can be rewritten as one.

🔄 for loop
# Square each level
squares = []
for n in [1,2,3,4,5]:
    squares.append(n ** 2)
# [1, 4, 9, 16, 25]
✨ comprehension
# Same result, one line
squares = [n ** 2
          for n in [1,2,3,4,5]]
# [1, 4, 9, 16, 25]
[ expression  for  var  in  iterable ]
expressionwhat to put in the list
varloop variable
iterableanything you can loop over
# Examples with RPG data
names  = ["aria", "brom", "caela"]
upper  = [n.upper() for n in names]
# ["ARIA", "BROM", "CAELA"]

levels = [1, 5, 12, 20]
xp     = [lvl * 100 for lvl in levels]
# [100, 500, 1200, 2000]

Filtering with if

~10 min

Add an if condition at the end to filter which items make it into the new list. Items that don't pass the condition are skipped entirely.

[ expression  for  var  in  iterable  if condition ]
if conditionskip if False
# Only keep heroes with level >= 10
party = [
    {"name": "Aria",  "level": 15},
    {"name": "Brom",  "level": 7},
    {"name": "Caela", "level": 22},
    {"name": "Dex",   "level": 3},
]

veterans = [h["name"] for h in party if h["level"] >= 10]
# ["Aria", "Caela"]

# Filter AND transform in one shot:
high_dmg = [d * 2 for d in [5, 18, 3, 27, 9] if d > 10]
# [36, 54]  — only 18 and 27 passed the filter

Nested Comprehensions

~10 min

You can nest a for clause to iterate over a 2D structure (a list of lists), or to combine two sequences — just like nested for loops.

🔄 nested for loops
grid = []
for row in [[1,2],[3,4]]:
  for n in row:
    grid.append(n)
# [1, 2, 3, 4]
✨ nested comprehension
grid = [n
        for row in [[1,2],[3,4]]
        for n in row]
# [1, 2, 3, 4]
# All (attacker, defender) matchup pairs
team_a = ["Aria", "Brom"]
team_b = ["Slag", "Vorx"]

matchups = [(a, b) for a in team_a for b in team_b]
# [("Aria","Slag"), ("Aria","Vorx"), ("Brom","Slag"), ("Brom","Vorx")]

# Flatten a dungeon map (list of row lists)
dungeon = [[0,1,0], [1,1,0], [0,0,1]]
walls   = [cell for row in dungeon for cell in row if cell == 1]
# [1, 1, 1, 1]  — only wall tiles
⚠️ Readability first: If a nested comprehension takes more than a glance to parse, use a regular loop instead. Clever ≠ clear.

When to Use vs. When to Loop

~8 min

List comprehensions are fast and expressive — but they aren't always the right tool. Use this guide:

SituationUse…Why
Build a new list from transform/filterComprehension ✓Concise, readable, fast
Logic has 3+ conditions or branchesfor loop ✓Easier to read and debug
You need to break/continue mid-loopfor loop ✓No break/continue in comprehensions
Side effects only (print, file write)for loop ✓Comprehensions shouldn't cause side effects
Nested more than 2 levels deepfor loop ✓Too hard to read quickly
Refactoring existing loop codeComprehension ✓Great opportunity to simplify
 
# ✅ Great comprehension candidate
alive = [h for h in party if h["hp"] > 0]

# ❌ Don't force it — use a loop instead
result = [
    process(h) if h["level"] > 10
    else fallback(h) if h["hp"] > 0
    else None
    for h in party if h["name"] != "Dex"
]  # ← readable? no. use a for loop.
