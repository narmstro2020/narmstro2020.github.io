📝 File I/O & Multithreading

L06 — Guided Notes · Apr 7, 2026 · 5 pts

Part 1 — Why File I/O?

Everything your program stores in variables disappears when the program ends. File I/O (Input/Output) lets you persist data — saving it to a file so it survives between runs.

🎮 Platformer connection: File I/O is how your game will save high scores, load level data, and store player settings between sessions.

The Two Directions

  • Input (Reading): File → Program — loading data from a file into your code
  • Output (Writing): Program → File — saving data from your code into a file

Section 1 — Reading Files with BufferedReader

Java uses a "wrapper" pattern for I/O. You wrap a FileReader (reads raw characters) inside a BufferedReader (adds buffering and the readLine() method).

try (BufferedReader reader = new BufferedReader(
        new FileReader("scores.txt"))) {

    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }

} catch (IOException e) {
    e.printStackTrace();
}

Key Details

  • readLine() returns one line at a time, or null when the file ends
  • The while loop reads until null — a standard Java I/O pattern
  • Both FileReader and BufferedReader live in java.io

Section 2 — try-with-resources

Notice the try (...) syntax — this is try-with-resources. Any object declared inside the parentheses that implements AutoCloseable is automatically closed when the block ends, even if an exception is thrown.

Why It Matters

Files are system resources. If you open a file and forget to close it (or an exception skips your .close() call), you create a resource leak. Try-with-resources guarantees cleanup.

Old Way (avoid this):

BufferedReader reader = null;
try {
    reader = new BufferedReader(new FileReader("data.txt"));
    // ... use reader ...
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (reader != null) reader.close(); // messy!
}

New Way (use this):

try (BufferedReader reader = new BufferedReader(
        new FileReader("data.txt"))) {
    // ... use reader ...
} catch (IOException e) {
    e.printStackTrace();
}
// reader is automatically closed here — even if an exception occurred

Section 3 — Writing Files with BufferedWriter

Writing follows the same wrapper pattern: FileWriter inside BufferedWriter. You can also use PrintWriter for convenience methods like println().

BufferedWriter

try (BufferedWriter writer = new BufferedWriter(
        new FileWriter("highscores.txt"))) {

    writer.write("Player1: 9500");
    writer.newLine();
    writer.write("Player2: 8200");
    writer.newLine();

} catch (IOException e) {
    e.printStackTrace();
}

PrintWriter (more convenient)

try (PrintWriter pw = new PrintWriter(
        new FileWriter("highscores.txt"))) {

    pw.println("Player1: 9500");
    pw.println("Player2: 8200");

} catch (IOException e) {
    e.printStackTrace();
}

⚠️ FileWriter overwrites by default! To append instead, pass true as the second argument: new FileWriter("file.txt", true)

Section 4 — Structured Data with JSON & Gson

Plain text files work for simple data, but games need structured data — objects with fields. JSON (JavaScript Object Notation) is a lightweight format, and Gson is a Google library that converts Java objects ↔ JSON automatically.

A JSON File (playerdata.json)

{
  "name": "Mario",
  "lives": 3,
  "score": 14500,
  "currentLevel": "world-1-2"
}

Java Object → JSON (Serialization)

Gson gson = new GsonBuilder().setPrettyPrinting().create();
String json = gson.toJson(playerData);

try (FileWriter fw = new FileWriter("playerdata.json")) {
    gson.toJson(playerData, fw);
}

JSON → Java Object (Deserialization)

try (FileReader fr = new FileReader("playerdata.json")) {
    PlayerData data = gson.fromJson(fr, PlayerData.class);
    System.out.println(data.getName());  // "Mario"
}

🎮 Platformer connection: You'll use Gson to save/load your game state — player position, lives, score, and current level — to a JSON file.

Part 2 — Why Multithreading?

By default, your Java program runs on a single thread — one instruction at a time, in order. Multithreading lets your program do multiple things simultaneously.

🎮 Platformer connection: libGDX's render loop runs on a single thread. If you load a large tilemap on that thread, the game freezes until loading finishes. Threads let you load assets in the background while showing a loading screen.

Real-World Examples

  • Loading assets in the background while showing a progress bar
  • A web server handling thousands of requests at once
  • Your computer running Spotify, IntelliJ, and Chrome simultaneously

Section 5 — Creating Threads

There are two main ways to create a thread in Java:

Option 1: Lambda with Thread (preferred)

Thread t = new Thread(() -> {
    System.out.println("Running on: " +
        Thread.currentThread().getName());
});
t.start();  // starts a NEW thread — don't call run() directly!

Option 2: Implementing Runnable

public class AssetLoader implements Runnable {
    @Override
    public void run() {
        System.out.println("Loading textures...");
        // ... heavy work here ...
    }
}

Thread loader = new Thread(new AssetLoader());
loader.start();

⚠️ start() vs run(): Calling start() creates a new thread and runs the code on it. Calling run() directly just executes the code on the current thread — no multithreading happens!

Section 6 — ExecutorService (Thread Pools)

Creating individual Thread objects is fine for simple cases, but in real applications you use an ExecutorService — a managed pool of reusable threads.

import java.util.concurrent.*;

ExecutorService pool = Executors.newFixedThreadPool(3);

pool.submit(() -> loadTextures());
pool.submit(() -> loadSounds());
pool.submit(() -> loadLevelData());

pool.shutdown();  // no new tasks, finish current ones

Why Use a Pool?

  • Reusable: Threads are expensive to create — a pool reuses them
  • Bounded: newFixedThreadPool(3) limits to 3 threads, preventing resource overload
  • Managed: The pool handles scheduling and lifecycle for you

Section 7 — Synchronized & Thread Safety

When multiple threads access the same variable, things can go wrong. This is called a race condition.

The Problem

private int score = 0;

// Thread A and Thread B both call this at the same time:
public void addPoints(int points) {
    score = score + points;  // READ score, ADD, WRITE back
}
// If both read score=100 simultaneously, both write 110
// Final result: 110 instead of 120 — a lost update!

The Fix: synchronized

public synchronized void addPoints(int points) {
    score = score + points;
}
// Now only one thread can enter this method at a time

The synchronized keyword acts like a lock on the method. Only one thread can hold the lock at a time — others must wait their turn.

Section 8 — The Game Thread in libGDX

libGDX runs your game on a single thread — the GL thread (or render thread). The render() method is called ~60 times per second on this thread.

Rules of the GL Thread

  • All rendering must happen on the GL thread — you cannot draw from a background thread
  • Never block the GL thread — file loading, network calls, or Thread.sleep() will freeze the game
  • AssetManager solves this — it loads assets on a background thread and hands them to the GL thread when ready
// In your game's render() method:
if (assetManager.update()) {
    // All assets loaded — switch to GameScreen
    setScreen(new GameScreen());
} else {
    // Still loading — show progress
    float progress = assetManager.getProgress();
    drawLoadingBar(progress);
}

🎮 Platformer connection: You'll use AssetManager to load textures, sounds, and tilemaps without freezing your game. This is multithreading in action — AssetManager uses a background thread internally.

Summary — Key Takeaways

  • BufferedReader / BufferedWriter — wrap raw readers/writers for efficiency and convenience
  • try-with-resources — automatically closes resources, prevents leaks
  • Gson — converts Java objects ↔ JSON for structured data persistence
  • Thread / Runnable — run code on a separate thread with start()
  • ExecutorService — managed thread pool for running multiple tasks
  • synchronized — prevents race conditions when threads share data
  • libGDX GL thread — never block it; use AssetManager for background loading
