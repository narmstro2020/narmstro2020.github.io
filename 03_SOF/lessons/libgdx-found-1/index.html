📝 Animation & Sprite Sheets

L07 — Guided Notes · Apr 13, 2026 · 5 pts · Final Project Prep

Why Animation?

A static sprite doesn't feel alive. Animation turns a single frozen image into a character that runs, jumps, and reacts. In libGDX, animation is just a sequence of still images played in rapid succession — like a flipbook.

🎮 Platformer connection: Your player will need at minimum 3 animations — idle, run, and jump. Enemies and collectibles will animate too.

Section 1 — Sprite Sheets

A sprite sheet is a single image file containing multiple animation frames laid out in a grid. Instead of loading 8 separate PNGs, you load one and slice it up in code.

Why a Sheet Instead of Individual Files?

  • Performance: One texture upload to the GPU instead of many
  • Memory: No per-file overhead
  • Batch rendering: libGDX can draw all frames from one texture in a single GPU call
  • Organization: All frames of one animation stay together

Typical Layout

Frames are usually laid out in a grid — each cell is the same size (e.g. 64×64 pixels). Rows often represent different animations (row 0 = idle, row 1 = run, row 2 = jump).

Section 2 — TextureRegion & split()

A TextureRegion is a rectangular slice of a larger Texture. It tells libGDX: "draw only this portion of the image." The static method TextureRegion.split() chops a sheet into a 2D array automatically.

Texture sheet = new Texture("player.png");  // 256×192 image

// Slice into 64×64 cells → returns TextureRegion[3][4]
TextureRegion[][] grid = TextureRegion.split(sheet, 64, 64);

TextureRegion[] idleFrames = grid[0];  // row 0
TextureRegion[] runFrames  = grid[1];  // row 1
TextureRegion[] jumpFrames = grid[2];  // row 2

⚠️ split() returns a 2D array: [rows][cols]. Rows first, then columns — a common source of off-by-one confusion.

Section 3 — The Animation Class

libGDX provides a generic Animation<T> class. For sprite animations, T is TextureRegion. You pass it the frame duration and the array of frames.

import com.badlogic.gdx.graphics.g2d.Animation;
import com.badlogic.gdx.graphics.g2d.TextureRegion;

Animation<TextureRegion> runAnim =
    new Animation<>(0.1f, runFrames);

runAnim.setPlayMode(Animation.PlayMode.LOOP);

Frame Duration

The first argument is seconds per frame, not total duration. Smaller = faster animation.

  • 0.1f → 10 frames per second (smooth walking)
  • 0.05f → 20 fps (fast run)
  • 0.25f → 4 fps (slow idle breathing)

Play Modes

  • NORMAL — plays once, stays on last frame (good for death)
  • LOOP — restarts from frame 0 forever (good for idle/run)
  • LOOP_PINGPONG — 0→N→0→N (good for hovering enemies)
  • LOOP_REVERSED — plays backwards repeatedly

Section 4 — stateTime: The Animation Clock

An Animation object is stateless — it doesn't know how long it's been playing. You track time in a field called stateTime, adding delta every frame.

private float stateTime = 0f;

@Override
public void render(float delta) {
    stateTime += delta;  // accumulate time

    TextureRegion currentFrame =
        runAnim.getKeyFrame(stateTime, true);
    // second arg: looping? (overrides PlayMode if set)

    batch.begin();
    batch.draw(currentFrame, player.x, player.y);
    batch.end();
}

💡 Why it works: getKeyFrame(stateTime) returns the frame whose duration range contains stateTime. If frames are 0.1s each and stateTime=0.25s, you get frame 2.

Section 5 — Switching Animations with State

A real character has multiple animations. Which one plays depends on what the character is doing. Use an enum to track state.

enum State { IDLE, RUNNING, JUMPING, FALLING }

State state = State.IDLE;

// Each frame, decide which state the player is in:
if (velocity.y > 0)        state = State.JUMPING;
else if (velocity.y < 0)  state = State.FALLING;
else if (velocity.x != 0) state = State.RUNNING;
else                       state = State.IDLE;

// Pick the right animation based on state
Animation<TextureRegion> current = switch (state) {
    case IDLE    -> idleAnim;
    case RUNNING -> runAnim;
    case JUMPING -> jumpAnim;
    case FALLING -> fallAnim;
};

⚠️ When switching animations, consider resetting stateTime = 0 so the new animation starts from frame 0 instead of mid-way through.

Section 6 — Flipping for Direction

You usually draw characters facing right by default. To face left, flip the texture horizontally — no need for duplicate left-facing art.

boolean facingRight = true;

if (Gdx.input.isKeyPressed(Keys.LEFT))  facingRight = false;
if (Gdx.input.isKeyPressed(Keys.RIGHT)) facingRight = true;

TextureRegion frame = runAnim.getKeyFrame(stateTime, true);

// Flip if needed — isFlipX() prevents double-flipping
if (!facingRight && !frame.isFlipX()) frame.flip(true, false);
else if (facingRight && frame.isFlipX()) frame.flip(true, false);

batch.draw(frame, player.x, player.y);

Summary — Key Takeaways

  • Sprite sheet — one image, many frames, one GPU upload
  • TextureRegion.split() — slices the sheet into a 2D array of regions
  • Animation<TextureRegion> — wraps frames + duration + play mode
  • stateTime — you track elapsed time, libGDX picks the right frame
  • PlayMode — NORMAL for one-shot, LOOP for repeating, PINGPONG for back-and-forth
  • State enum — cleanly switch between idle/run/jump animations
  • flip() — mirror a frame horizontally for left/right direction
